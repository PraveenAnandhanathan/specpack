## Outline

You are performing an AI-assisted performance report analysis. Your goal is to produce `profiles/performance-profile.md` — a set of concrete performance baselines and implementation constraints that all future work must respect.

### Step 1 — Locate reports

- If user provided a path, use that directory.
- Check if `profiles/.performance-analyse-context.md` exists and read the source path from it.
- Scan the directory for: `.csv`, `.json`, `.yaml`, `.yml`, `.txt`, `.log` files.
- Also look for: JMeter result files (`*.jtl`), Gatling reports (`simulation.log`), k6 output, Locust CSVs, APM exports (Datadog, New Relic, Dynatrace JSON), lighthouse reports.

### Step 2 — Read and parse reports

For each file found:
1. Determine the tool that generated it (JMeter, Gatling, k6, Locust, Lighthouse, APM, custom).
2. Extract key metrics:
   - **Latency / Response time**: min, mean, median (P50), P90, P95, P99, max
   - **Throughput**: requests/sec, transactions/sec
   - **Error rate**: % errors, types of errors
   - **Concurrency**: virtual users, concurrent connections
   - **Resource utilisation**: CPU %, memory %, if present
   - **Apdex score** if present
3. Note the test scenario: load test, stress test, spike test, soak test — if detectable.
4. Note date/version if present.

### Step 3 — Synthesise baselines

Across all reports:
- Identify the **baseline (normal load)** metrics — the P50/P90 at expected concurrent users.
- Identify the **peak load** metrics — max concurrency tested and resulting P95/P99.
- Identify **bottlenecks** mentioned or implied by high P99 / error spikes.
- Note any **SLAs or targets** if explicitly mentioned in report file names or metadata.
