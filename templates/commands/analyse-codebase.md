## Outline

You are performing an AI-assisted codebase analysis. Your goal is to produce a rich `profiles/codebase-profile.md` that the AI agents will use as a coding style reference for all future spec, plan, and implementation work.

**Important**: Read broadly but write concisely. The profile is a reference document for AI agents, not a report for humans.

### Step 1 — Determine the analysis root

- If user provided a path or URL, use that.
- Check if `.profiles/.codebase-analyse-context.md` exists — if so, read it for the source path.
- Otherwise, analyse the current working directory.

### Step 2 — Discover languages and file structure

1. Walk the directory tree (skip: `node_modules`, `__pycache__`, `.git`, `dist`, `build`, `.venv`, `venv`, `.specify`, `profiles`).
2. Count file extensions and map to languages.
3. Capture the top-level and second-level directory structure (max 3 levels deep).
4. Identify entry points: `main.*`, `index.*`, `app.*`, `server.*`, `cli.*`, `__init__.py`, etc.

### Step 3 — Detect frameworks and tooling

Check for presence of:
- **Package managers**: `package.json`, `requirements.txt`, `pyproject.toml`, `Cargo.toml`, `go.mod`, `pom.xml`, `build.gradle`, `Gemfile`, `composer.json`
- **Test frameworks**: `pytest.ini`, `jest.config.*`, `vitest.config.*`, `.rspec`, `phpunit.xml`, `karma.conf.*`
- **Code style / linting**: `.eslintrc*`, `eslint.config.*`, `.prettierrc*`, `.flake8`, `pyproject.toml [tool.ruff]`, `.editorconfig`
- **CI/CD**: `.github/workflows/`, `.gitlab-ci.yml`, `Jenkinsfile`, `azure-pipelines.yml`
- **Containers**: `Dockerfile`, `docker-compose.yml`
- **IaC**: `terraform.tf`, `helm/`, `k8s/`

### Step 4 — Sample code for style patterns

Read 3–5 representative source files (pick the most complex non-test files in the dominant language). Identify:
- **Naming conventions**: camelCase / snake_case / PascalCase for variables, functions, classes
- **Module structure**: how imports are organized, how modules are split
- **Error handling style**: exceptions vs return codes, logging patterns
- **Comment style**: docstrings, inline comments, JSDoc, etc.
- **Test structure**: test file naming, test organisation pattern (describe/it, class-based, etc.)
- **Async patterns**: callbacks, promises, async/await, coroutines
- **Dependency injection / architecture pattern** (if detectable)

### Step 5 — Detect existing test framework (critical for validation commands)

Search all MD files in `.specify/` for mentions of test frameworks (Jest, pytest, JUnit, RSpec, Vitest, Mocha, etc.). Also check config files. Record the **canonical test command** if detectable (`npm test`, `pytest`, `./gradlew test`, etc.).
